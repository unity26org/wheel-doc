Default Resources

  • Sessions
  • Myself
  • Users

When you create a new API with Wheel, its generates, by default, three resources: Session, Myself and Users. Each one has its own routes (check file /app/root/path/routes/routes.go) and access permissions (check file /app/root/path/routes/authorization.go). And they can share functionalities from theirs handlers, models and views.

To access the API resources, we suggest that you to use the Postman client. But feel free to choose another similar tool.

If you didn't change, the default credentials of the admin user are:

email user@example.com
password !Secret.123!

You are gonna to have these credentials to sign in as an admin user. Make sure you already have ran the migration.

$> go run main.go -mode=migrate

Sessions

Session manages user's access to the API. Its services are: sign up, sign in, sign out, password recovery and session refresh.

Sign Up

The public of the internet can sign up, but they can't sign up as admin. Only an admin can create admins. You are going to see how in Users section.

POST /sessions/sign_up

Name Description HTTP Type Mandatory
name New user's name Body String Yes
email New user's email Body String Yes
password New user's password Body String Yes
locale New user's locale. Available locales are "pt-BR" and "en" Body String Yes
On success: 

{
    "system_message": {
        "type": "notice",
        "content": "signed in successfully"
    },
    "token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9....",
    "expires": 7200
}
On error: 

{
    "system_message": {
        "type": "alert",
        "content": "user was not created"
    },
    "errors": [
    ...
    ]
}
Avaiable errors are:
  • name can't be blank
  • name is too long
  • email can't be blank
  • email is too long
  • email is invalid
  • email has already been taken
  • password is too short minimum is 8 characters
  • locale is invalid

Sign In

Users with a valid credential can sign in the system. Upon successful sign in, the user will recieve a JWT-based token and an expiration time in seconds. It means that you need to provide the token in furthers requests to get access. Be aware to update the token before it get expired calling the session refresh service.

POST /sessions/sign_in

Name Description HTTP Type Mandatory
email User's email Body String Yes
password User's password Body String Yes
On success: 

{
    "system_message": {
        "type": "notice",
        "content": "signed in successfully"
    },
    "token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9....",
    "expires": 7200
}
On error: 

{
    "system_message": {
        "type": "alert",
        "content": "could not sign in"
    },
    "errors": ["invalid credentials"]
}

Sign Out

Sign out for signed in users.

DELETE /sessions/sign_out

Name Description HTTP Type Mandatory
token Session token Header String Yes
On success: 

{
  "system_message": {
    "type":"notice",
    "content":"signed out successfully"
  }
}
On error: 

{
    "system_message": {
        "type": "alert",
        "content": "access denied"
    },
    "errors": ["invalid token"]
}

Request Password Recovery

Kick off to password recovery

POST /sessions/password

Name Description HTTP Type Mandatory
email User's email' Body String Yes
On success or on error: 

{
    "system_message": {
        "type": "notice",
        "content": "user password recovery instructions was successfully sent"
    }
}
              

To avoid user enumeration vulnerability there's not error message.

Password Recovering

Confirm password recovery

PUT /sessions/password

Name Description HTTP Type Mandatory
token Recovery passoword token Body String Yes
new_password New password Body String Yes
password_confirmation Password confirmation Body String Yes
On success: 

{
    "system_message": {
        "type": "notice",
        "content": "password was successfully changed"
    }
}
On error: 

{
    "system_message": {
        "type": "alert",
        "content": "password could not be changed"
    },
    "errors": [
    ...
    ]
}
Avaiable errors are:
  • invalid reset password token
  • password confirmation does not match new password
  • password is too short minimum is 8 characters

Session Refresh

Refresh the session. Returns a new JWT-based token and a new expiration time in seconds.

POST /sessions/sign_refresh

Name Description HTTP Type Mandatory
token Session token Header String Yes
On success: 

{
    "system_message": {
        "type": "notice",
        "content": "session was successfully refreshed"
    },
    "token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
    "expires": 7200
}
On error: 

{
    "system_message": {
        "type": "alert",
        "content": "access denied"
    },
    "errors": ["invalid token"]
}

Myself

Myself are services whose the users can read and update their own personal data. Also users can change their password.

Read

Returns current user's personal data.

GET /myself

Name Description HTTP Type Mandatory
token Session token Header String Yes
On success: 

{
    "id": 1,
    "name": "User Name",
    "email": "user@example.com",
    "created_at": "2019-08-08T01:21:37.351901Z",
    "updated_at": "2019-08-08T01:21:37.351901Z"
}
On error: 

{
    "system_message": {
        "type": "alert",
        "content": "access denied"
    },
    "errors": ["invalid token"]
}
